## 理论-案例闭环

```
理论文档                案例文档
┌─────────┐            ┌─────────┐
│ 原理A   │───────────→│ 案例1   │
│         │            │ 案例2   │
└─────────┘            └─────────┘
     ↑                      │
     └──────验证与反哺───────┘
```

- 每个原理必须链接到验证案例
- 每个案例必须标注使用的原理
- 双向验证：原理指导实践，案例验证原理

## 处理原则

1. **不要表面技巧，要底层原理**
2. **不要"怎么说"，要"为什么这样说有效"**
3. **不要话术模板，要心理机制**
4. **不要操作步骤，要设计哲学**
5. **每个方法背后都问：这利用了人的什么心理/认知规律？**

### 示例对比

| 错误做法 | 正确做法 |
|----------|----------|
| 使用"限时优惠"话术 | 利用损失厌恶心理——人对失去的恐惧是获得快乐的2倍 |
| Agent要有明确的工具定义 | 工具定义本质是"契约设计"——在确定性系统和非确定性LLM之间建立可靠的通信协议 |
| "仅剩3件库存"文案 | 稀缺效应+竞争紧迫感——激活损失厌恶，让用户从"要不要买"转为"赶紧买" |

## 案例处理提示词

针对案例类URL，使用以下提示词：

```
你是一个案例分析专家。请从这篇内容中提取案例信息。

重点提取：
1. 案例背景：他们面临什么问题？
2. 解决方案：具体做了什么？
3. 效果数据：有什么量化结果？
4. 底层原理：这个方案利用了什么心理学/行为学原理？

如果原文没有明确的原理分析，请根据做法推断可能运用的原理。

关联到这些原理类别：
- 损失厌恶、互惠原则、承诺一致性、社会认同
- 锚定效应、稀缺效应、禀赋效应、框架效应
- 认知闭合、心理抗拒、默认效应
```

## 质量检验

每篇文档问自己：
1. 读完能理解"为什么"吗？
2. 原理能迁移到其他场景吗？
3. 三年后这个知识还有用吗？
4. 理论文档是否链接了案例？案例是否关联了理论？
